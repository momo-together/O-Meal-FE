# 기능 구현 가이드

## 작업 시작 전 체크

이슈에 명시된 항목을 먼저 확인한다.

- 상태 관리 방식 → 명시된 경우 해당 방식으로 구현
- 에러 처리 방식 → 명시된 경우 해당 방식으로 구현
- 테스트 작성 범위 → 명시된 경우 해당 범위로 작성, 없으면 도메인 로직 전체

---

## 설계 원칙

### 확장성 우선

기능을 구현할 때 다음 질문을 기준으로 설계한다:

- 다음에 유사한 기능이 추가된다면 이 구조가 그대로 대응 가능한가?
- 지금 하드코딩된 부분이 나중에 변경될 가능성이 있는가?
- 의존성이 특정 구현에 묶여 있지는 않은가?

애매한 설계 결정이 있었다면 PR 본문에 반드시 고려한 내용을 남긴다.

---

## 코드 설계 기준

### 훅 분리

UI 로직과 비즈니스 로직을 분리한다.

```typescript
// ✅ 분리된 구조
const useFeatureName = () => {
  // 비즈니스 로직, 상태 관리
  return { data, handlers };
};

const FeatureComponent = () => {
  const { data, handlers } = useFeatureName();
  return <div>...</div>;
};
```

### 타입 설계

- 확장 가능한 타입 구조를 설계할 것
- `any` 사용 금지 (TypeScript strict mode 준수)
- 공통으로 쓰이는 타입은 별도 파일로 분리

## TDD 사이클 (도메인 로직)

비즈니스 로직, 유틸리티 함수, 커스텀 훅은 TDD로 개발한다.

**Red → Green → Refactor**

1. 실패하는 테스트 먼저 작성
2. 테스트를 통과하는 최소한의 코드 작성
3. 리팩토링

### 커버리지 목표

- 도메인 로직: 90% 이상
- 유틸리티 함수: 100%
  목표 커버리지에 도달하지 못한 경우 최대 3회까지 재시도한다. 3회 후에도 달성하지 못하면 PR에 아래 형식으로 남긴다:

```
// TODO: 커버리지 미달 - 현재 XX%, 목표 XX%
// 미달 이유: (구체적인 이유)
// 미달 항목: (커버되지 않은 케이스)
```

---

## 빌드/타입 에러 발생 시

1. 에러 원인 파악 후 직접 수정 시도
2. 수정이 불가능하거나 설계 변경이 필요한 경우 → PR에 TODO 주석과 함께 이유 명시
3. 빌드가 깨진 상태로 PR을 올리지 않는다

```typescript
// TODO: [이슈번호] 에러 내용 및 해결 필요 이유
```

---

## PR 크기 제한

- 한 PR = 한 이슈
- 기능 구현 중 범위가 커진다고 판단되면 PR 본문에 별도 이슈 제안을 남길 것
- 리팩토링이 필요한 부분이 보이더라도 별도 이슈로 분리할 것

---

## PR 작성 시 포함할 내용

- 구현 방식 선택 이유
- 확장성을 고려한 설계 결정 사항
- 애매했던 부분과 그에 대한 판단 근거
- 추가로 파생될 수 있는 작업이 있다면 제안

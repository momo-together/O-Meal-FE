# PR 작성 가이드

## 기본 원칙

- 한국어로 작성
- 문장 호흡을 짧게 (모바일 가독성 우선)
- 한 PR = 한 이슈

---

## PR 제목

```
[type] #이슈번호 작업 내용 요약
```

type 종류:

- `feat`: 기능 추가
- `fix`: 버그 수정
- `refactor`: 리팩토링
- `chore`: 설정, 의존성 등
- `docs`: 문서

예시:

```
[feat] #12 식당 목록 필터 컴포넌트 구현
[fix] #34 좋아요 버튼 중복 클릭 방지
```

---

## PR 본문

작업 유형에 따라 해당 가이드 문서의 'PR 작성 시 포함할 내용' 섹션을 반드시 참고한다.

| 작업 유형        | 참고 문서                        |
| ---------------- | -------------------------------- |
| UI/컴포넌트 구현 | `.claude/guides/ui-component.md` |
| 기능 구현        | `.claude/guides/feature.md`      |

### 공통 템플릿

```markdown
## 작업 내용

<!-- 무엇을 했는지 짧게 -->

## 변경 사항

<!-- 변경된 파일/기능 목록 -->

## 설계 결정

<!-- 확장성, 구조 선택 이유 등 애매했던 부분 -->

## 스크린샷

<!-- UI 작업이라면 첨부 -->

## 체크리스트

- [ ] 빌드 통과
- [ ] 타입 에러 없음
- [ ] 테스트 작성 (도메인 로직)
- [ ] Storybook 작성 (컴포넌트)
- [ ] 접근성 처리

Closes #이슈번호
```

## 코드 코멘트

코드 단위 설명이 필요한 경우 PR 코멘트로 해당 라인에 직접 남긴다.

대상:

- 직관적이지 않은 로직
- 의도적으로 특이하게 구현한 부분
- 추후 변경 가능성이 있는 부분

형식:

```
이 부분은 [이유] 때문에 [방식]으로 구현했습니다.
추후 [상황]이 생기면 [방향]으로 변경이 필요합니다.
```

---

## 파생 작업 제안

기능 구현 중 추가 작업이 필요하다고 판단되면 PR 본문 하단에 명시:

```markdown
## 파생 작업 제안

- [ ] [내용] → 별도 이슈 필요
```

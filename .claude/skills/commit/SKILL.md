# 커밋 스킬 (Commit Skill)

스테이지된 변경사항을 분석하여 커밋 단위와 메시지를 **자동으로 결정**하고, 사용자 확인 없이 **커밋까지 바로 실행**합니다.

## 주의사항

- **절대 `git push`를 실행하지 않습니다.** push는 어떤 상황에서도 금지입니다.
- 사용자에게 묻지 않고 분석 → 커밋까지 자동 실행합니다.

## 작동 방식

1. 스테이지된 변경사항 확인
2. 커밋 단위 및 메시지 자동 결정
3. 커밋 순서대로 즉시 실행
4. 완료 결과 출력

## 커밋 메시지 형식

```
{타입}: {제목}

{본문 - 상세 설명 (필요한 경우만)}
```

## 커밋 타입 기준

| 타입     | 적용 기준                                  |
| -------- | ------------------------------------------ |
| feat     | 새로운 기능 추가, 기존 로직 변경 및 개선   |
| fix      | 의도한 동작과 다르게 구현된 버그 수정      |
| refactor | 동작은 동일하나 코드 구조/내부 로직 변경   |
| style    | 로직 변경 없이 포맷, CSS 등 스타일만 변경  |
| docs     | 주석 추가, 오타 수정, 문서 변경            |
| test     | 테스트 코드 추가 및 수정                   |
| chore    | 빌드 설정, 패키지, 환경 설정 등 그 외 작업 |

**타입 결정 우선순위**: 변경 목적이 불분명할 경우 `feat > fix > refactor` 순으로 판단합니다.

## 커밋 단위 분리 기준

다음 조건 중 하나라도 해당하면 자동으로 분리합니다:

1. 서로 다른 타입의 변경이 섞여 있는 경우 (예: feat + fix + refactor)
2. 여러 도메인/폴더에 걸친 변경이 있는 경우
3. 커밋 제목을 한 문장으로 설명하기 어려운 경우

## 실행 순서

### 1. 변경사항 분석 및 커밋 계획 출력

분석 결과와 실행할 커밋 목록을 출력합니다:

```
📦 커밋 계획:

[1/3] feat: 상품 목록 필터 기능 추가
  파일: src/components/ProductFilter.tsx, src/utils/filterHelpers.ts

[2/3] fix: 장바구니 수량 변경 시 합계 미반영 수정
  파일: src/components/CartItem.tsx

[3/3] refactor: 공통 API 호출 로직 훅으로 분리
  파일: src/hooks/useFetch.ts
```

### 2. 커밋 즉시 실행

계획 출력 직후 순서대로 실행합니다:

1. 분석 시점의 staged 상태를 기준으로 커밋 대상 확정
2. 필요 시 `git commit --only <files>` 또는 hunk 단위 staging(`git add -p`)으로 범위 고정하여 unstaged 변경이 포함되지 않도록 주의
3. `git commit` 실행
4. 모든 커밋 완료 후 결과 출력

```
✅ 모든 커밋이 완료되었습니다!

[1/3] feat: 상품 목록 필터 기능 추가 ✓
[2/3] fix: 장바구니 수량 변경 시 합계 미반영 수정 ✓
[3/3] refactor: 공통 API 호출 로직 훅으로 분리 ✓
```

**⚠️ `git push`는 절대 실행하지 않습니다. push가 필요하면 사용자가 직접 실행하세요.**

## 에러 처리

### 스테이지된 변경사항이 없는 경우

```
❌ 스테이지된 변경사항이 없습니다.
먼저 변경사항을 스테이지하세요.

git add <파일명>
# 또는
git add .
```
